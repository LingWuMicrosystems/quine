## Goal
Decompose `run_query` (~140 lines, lines 299-439) into a readable 3-stage pipeline by extracting three private helper functions, reducing the method body to ~35 lines.

## Purpose
Phase 11 simplified `related_egraph.rs` from 892 to 641 lines by extracting CostTracker, ReverseIndex, and deduplicating mutation paths. `run_query` remains the deepest-nested function in the file — a structural outlier. This refactor makes the query evaluation pipeline self-documenting: scan → join → filter/permute, matching how the Datalog semi-naive evaluation actually works.

## Output
- Three private methods on `RelatedEGraph`: `scan_step_table()`, `join_step_rows()`, `filter_and_permute()`
- Refactored `run_query` reduced to ~35-line readable pipeline
- Zero behavioral changes — 46/46 existing tests continue to pass
- `related_egraph.rs` line count should decrease modestly (extracted signatures add some overhead)

  <name>Extract scan_step_table — encapsulate per-step table scanning</name>
  <files>quine-core/src/related_egraph.rs</files>
  <action>
    Extract the scan logic for a single query step (lines 338-372) into:
    `fn scan_step_table(&self, step: &ScanStep, delta_table: Option<TableId>, filter: Option<(usize, &Set<Value>)>) -> Vec<Row>`

    This function handles BOTH scan paths:
    - **Plain scan:** when filter is None — single fused_scan call (the else branch, lines 362-372)
    - **Push-down scan:** when filter is Some((column, distinct_values)) — iterate distinct values, clone constraints, add Equ constraint per value, flat_map fused_scan results (lines 338-361)

    The filter parameter abstracts the "sideways information passing" optimization: when a previous step shares a variable with this step, the caller collects distinct values and passes them in.

    What to avoid:
    - Do NOT change the scan behavior — fused_scan arguments must be identical
    - Do NOT change how delta_table filters (Some(step.table) comparison)
    - Do NOT clone constraints unnecessarily in the plain scan path
  </action>
