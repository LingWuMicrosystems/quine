<objective>
## Goal
Define a cost lattice `L = (N, 0, u64::MAX, min)` and maintain eclass costs incrementally during `insert`, `union`, and `rebuild` operations on `RelatedEGraph`. Track which enode achieves the minimum cost per eclass (`cost_select`), and maintain correctness when enodes are absorbed during rebuild.

## Purpose
The e-graph is a dynamically-growing DAG under equality saturation. Cost computation must be incremental â€” recomputing from scratch after every `run` would be wasteful. By treating cost as a fixpoint over a lattice with monotonic decreases, we compute costs as a natural side-effect of e-graph maintenance.

## Output
- Lattice `(u64, min, u64::MAX)` integrated into `RelatedEGraph`
- `cost_models: Map<String, u64>` moved from `EngineContext` to `RelatedEGraph`
- `eclass_cost: Map<Value, u64>` â€” current min cost per canonical eclass (âŠĄ = u64::MAX)
- `cost_select: Map<Value, (TableId, RowIndex)>` â€” cheapest enode per eclass
- Cost maintained incrementally in `insert`, `union`, `rebuild`
- When enode is absorbed during rebuild, `cost_select` redirected to surviving `RowIndex`
- BDD integration tests
</objective>

## AC-1: Cost lattice definition
```gherkin
Given the cost lattice L = { u64 values, partial order: a âŠ‘ b iff a â‰Ą b, join âŠ” = min, bottom âŠĄ = u64::MAX, top âŠ¤ = 0 }
When  an eclass has no enodes (fresh allocation)
Then  its cost is u64::MAX (unknown/âŠĄ)
When  an eclass has at least one enode with computable cost
Then  its cost is the minimum among all its enodes' costs
```

## AC-2: Cost computed on insert
```gherkin
Given `cost Expr.Add = 1` and child eclasses already have costs 2 and 3
When  an Add enode is inserted with those children, producing result eclass E
Then  eclass_cost[E] = 1 + 2 + 3 = 6
And   cost_select[E] = (table_id_of_Expr.Add, row_index_of_inserted_enode)
```

## AC-3: Cost merged on union
```gherkin
Given eclass A has cost 5 with cost_select = (T1, R1)
And   eclass B has cost 3 with cost_select = (T2, R2)
When  union(A, B) is performed, producing parent eclass A
Then  eclass_cost[A] = min(5, 3) = 3
And   cost_select[A] = (T2, R2) (the cheaper enode)
And   eclass_cost and cost_select for B are removed
```

## AC-4: cost_select redirected when enode absorbed during rebuild
```gherkin
Given two Add enodes at (T, R_scanned) and (T, R_existing) with same canonical children
And   cost_select[E] = (T, R_scanned)
When  rebuild absorbs R_scanned into R_existing (reverse_index removes R_scanned, union merges result eclasses)
Then  cost_select for the merged result eclass points to (T, R_existing), not the absorbed row
```

## AC-5: Undefined constructor defaults to cost 0
```gherkin
Given an enode with table name "Expr.Add" but no entry in cost_models
When  the enode's cost is computed
Then  constructor_cost = 0
```

## AC-6: Child with unknown cost propagates u64::MAX
```gherkin

When  the enode's cost is computed
Then  enode_cost = u64::MAX (saturating_add propagates unknown)
```
