<objective>
## Goal
Add `Term::Let` and `Term::Var` variants to the `Term` enum, and implement two-pass (reference-counting + build) extraction so that eclasses referenced more than once in the extraction DAG are bound with `let` bindings instead of being duplicated in the printed output.

## Purpose
Currently, `extract` and `extract optimal` produce a `Term` tree via recursive DFS from the root eclass. If a sub-expression eclass is referenced by multiple parent enodes, the term for that eclass is duplicated in the output. This produces exponentially large output for DAG-structured expressions, making extraction results unreadable for non-trivial e-graphs.

`Term::Let` solves this by binding multiply-referenced eclasses to auto-generated names, emitting a flat `(let ([_t0 <binding>] [_t1 <binding>]) <body>)` form. All bindings are collected into a single `Let` node (not nested), avoiding the readability problem of deeply nested `let` forms. The resulting output is linear in the size of the extraction DAG.

## Output
- `Term::Let(Vec<(String, Term)>, Box<Term>)` variant — flat vector of `(name, binding)` pairs + body. All shared sub-expressions collected into a single Let node, avoiding deep nesting.
- `Term::Var(String)` variant — reference to a let-bound variable
- Updated `Display` impl:
  - `Term::Let(bindings, body)` → `(let ([name1 val1] [name2 val2]) body)` — square brackets around each `[name val]` pair
  - `Term::Var(name)` → bare name
- `count_eclass_refs(&self, eclass) -> Map<Value, usize>` — DFS pass counting how many enodes reference each eclass in the extraction path
- `materialize_cheapest_with_lets(&self, eclass) -> Term` — cost-aware build with let-introduction
- `extract_inner_with_lets(&self, id, ...) -> Term` — greedy build with let-introduction
- Both existing `materialize_cheapest` and `extract_inner` preserved (they remain useful for testing and simple cases)
- Integration tests verifying let-binding and non-duplication
</objective>

<task type="auto">
  <name>Task 2: Implement two-pass let-aware extraction</name>
  <files>quine-frontend/src/lib.rs</files>
  <action>
    Implement reference-counting + let-introduction for both extraction paths.

    **Pass 1 — Reference counting (`count_eclass_refs`):**
    - Follow the same DAG traversal as the extraction path (cheapest enode at each eclass)
    - For each enode visited, increment ref count for all child eclasses (type Name or Id)
    - Uses a `visited` set to avoid double-counting cycles
    - Returns `Map<Value, usize>` — eclass → number of parent enodes referencing it

    **Pass 2 — Build with lets (`build_term_with_lets`):**
    - Takes: eclass, ref_counts, bindings (Map<Value, String>), name_counter (&mut usize),
      pending_bindings (&mut Vec<(String, Term)>), visited
    - `pending_bindings` accumulates ALL shared-child bindings across the entire extraction
      (threaded by `&mut` through recursion)
    - Follows cheapest enode at each eclass
    - For each child position that is an eclass type:
      - If `ref_counts[child] > 1` (shared):
        - If child already in `bindings` → use `Term::Var(name.clone())`
        - If child NOT in bindings → generate name `_t{name_counter}`, increment counter,
          insert into bindings, recursively build binding term, push `(name, binding)` to
          `pending_bindings`, use `Term::Var(name)` as child
      - Else (not shared) → recurse normally, check `visited` for cycles
    - After building all children, return `Term::App(func_name, children)` — NO inline wrapping
    - The TOP-LEVEL `materialize_cheapest_with_lets` wraps the root term:
      ```rust
      if pending_bindings.is_empty() {
          root_term
      } else {
          Term::Let(pending_bindings, Box::new(root_term))
      }
      ```
    - Result: exactly ONE Let node at the top level, no nesting

    **Public API additions:**

    ```rust
    impl EngineContext {
        /// Count how many enodes reference each eclass in the extraction DAG.
        fn count_eclass_refs(&self, eclass: Value) -> Map<Value, usize>;

        /// Cost-aware materialization with let-bindings for shared eclasses.
        pub fn materialize_cheapest_with_lets(&self, eclass: Value) -> Term;

        /// Greedy extraction with let-bindings for shared eclasses.
        fn extract_inner_with_lets(&self, id: Value, bindings: &mut Map<Value, String>,
            name_counter: &mut usize, visited: &mut Set<Value>) -> Term;
    }
    ```

    **Wire into `evaluate_and_extract`:**
    - Replace `self.materialize_cheapest(eclass)` with `self.materialize_cheapest_with_lets(eclass)`
    - This automatically enables let-binding for both `extract` and `extract optimal`

    **Preserve existing methods:**
    - Keep `materialize_cheapest` and `extract_inner` unchanged — they remain useful for
      testing and for cases where let-binding is undesirable

    Avoid:
    - Modifying the existing `materialize_cheapest` signature (backward compat)
    - Using recursion without visited tracking (infinite loops on cycles)
    - Let-binding an eclass that is referenced only once (unnecessary indirection)
    - Nested Let nodes — collect all bindings into one Let via `&mut Vec` threading
  </action>
