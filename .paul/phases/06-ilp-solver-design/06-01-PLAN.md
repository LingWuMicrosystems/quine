<objective>
## Goal
Produce a comprehensive ILP Solver Design Report that designs a custom, purpose-built ILP solver algorithm for e-graph extraction, defines the ILP formulation mapped to Quine data structures, and specifies the solver architecture within the project — providing the blueprint for Phases 7 (Implementation), 8 (Integration), and 9 (Enhanced Extraction).

## Purpose
The current extraction system uses greedy local-minimum selection via `materialize_cheapest`. While correct for independent costs, it cannot account for shared subexpressions (CSE), global cost trade-offs, or multi-objective optimization. An ILP-based extraction would find globally optimal expressions. **Per project directive, the ILP solver must be implemented entirely within the Quine project — zero external solver libraries or tools.** The solver is purpose-built for e-graph extraction only, exploiting the specific problem structure for performance and minimal code size.

  <name>Task 1: Design the custom solver algorithm</name>
  <files>.paul/phases/06-ilp-solver-design/ILP-DESIGN-REPORT.md</files>
  <action>
    Design the custom ILP solver algorithm for e-graph extraction. No external solver libraries — the algorithm is implemented from scratch within the project.

    **Part A: Problem structure analysis**

    Characterize the e-graph extraction problem to identify exploitable structure:
    - The extraction graph is a DAG (eclasses as nodes, enodes as hyperedges from parent eclass to child eclasses)
    - The ILP constraint matrix is sparse (each enode references 0-3 child eclasses typically)
    - Problem sizes: 50-500 variables, 100-2000 constraints for typical extractions
    - Without CSE, the problem reduces to shortest-path on a DAG (greedy = optimal)
    - With CSE, the y_e variables couple otherwise-independent branches — this is the core difficulty
    - The constraint matrix has a block structure (each eclass is a block of x variables)

    **Part B: Algorithmic approaches to evaluate (minimum 3)**

    1. **Branch-and-bound with simplex LP relaxation**
       - LP relaxation: drop integrality, solve with revised simplex on sparse tableau
       - Branching: pick fractional x_{e,n}, branch on =0 vs =1
       - Node selection: best-bound first (small search tree on well-structured problems)
       - Bounds: LP optimum at each node, prune when ≥ incumbent
       - Exploit: the constraint matrix is sparse and near-triangular after ordering by DAG depth
       - Code estimate: 500-800 lines for simplex + 200-400 for B&B framework

    2. **Greedy + iterative improvement (local search)**
       - Start: greedy extraction (current `materialize_cheapest`) as initial feasible solution
       - Neighborhood: swap one eclass's selected enode, re-solve affected subtree
       - Acceptance: accept if cost decreases (hill-climbing) or use simulated annealing
       - Exploit: the DAG structure means local changes only affect ancestors
       - Limitation: no optimality guarantee, may miss globally optimal solutions
       - Code estimate: 200-400 lines

    3. **Dynamic programming with Lagrangian relaxation for CSE**
       - Without CSE: DP from leaves to root, O(|E| * avg_enodes) — same as greedy
       - With CSE: Lagrangian relaxation — penalize shared eclass cost discrepancies
       - Iterate: solve relaxed DP, update Lagrange multipliers, converge to near-optimal
       - Exploit: the problem is "almost" a tree; CSE edges are the few cross-links
       - Code estimate: 300-500 lines

    4. **Branch-and-bound with problem-specific relaxation** (bonus 4th approach)
       - Instead of general LP, use a combinatorial relaxation: drop CSE constraints, solve the resulting DAG shortest-path (which is trivial via DP)
       - This relaxation is much faster than simplex LP and potentially tighter for this problem
       - Branch on y_e variables (which eclasses are shared) — branching strategy exploits problem structure
       - Code estimate: 300-500 lines (no simplex needed)

    5. **Min-cost flow reduction attempt**
       - Investigate: can extraction without CSE be formulated as min-cost flow? (Yes — DAG shortest path.)
       - With CSE: does it become min-cost flow with bundling constraints? Possibly a fixed-charge flow problem.
       - Assessment: likely doesn't cleanly reduce; document the reduction attempt and its limits
       - If viable: polynomial time, extremely fast, very small code

    **Part C: Evaluation criteria**

    For each approach, assess:
    | Criterion | Weight | Description |
    |-----------|--------|-------------|
    | Optimality guarantee | High | Does it find the global optimum or only approximate? |
    | Expected runtime (50-500 vars) | High | Millisecond-level? Second-level? |
    | Code complexity (lines) | High | Must be maintainable within the project |
    | Exploits DAG structure | Medium | Does the algorithm use problem-specific structure? |
    | Worst-case behavior | Medium | Degenerate cases (wide DAG, deep DAG, many CSE edges) |
    | Incremental capability | Low | Can it reuse results when the e-graph grows? |

    **Part D: Recommendation**

    Produce a clear recommendation. Consider a staged approach:
    - Phase 7: implement the simplest algorithm with good average-case behavior
    - Phase 9: optionally upgrade to a more sophisticated algorithm if needed

    The recommendation must include:
    - Algorithm pseudocode (20-50 lines) sufficient to guide implementation
    - Key data structures (priority queue, tableau, DP table, etc.)
    - Where each structure lives in the Quine type system
    - Expected performance bounds on representative extraction problems

    **Key insight to exploit:**
    The extraction problem is a 0-1 ILP on a DAG where CSE coupling is the ONLY reason it's not a simple shortest-path. Most eclasses have 1-3 enodes. The solver should exploit this sparsity and near-tree structure rather than treating it as a general ILP.

    Avoid: designing a general-purpose ILP solver. The solver ONLY needs to handle the extraction ILP. Exploit every aspect of the problem structure — DAG topology, sparsity, small arity, few enodes per eclass, CSE as the only coupling constraint.
  </action>

  <name>Task 2: Design ILP formulation for e-graph extraction</name>
  <files>.paul/phases/06-ilp-solver-design/ILP-DESIGN-REPORT.md</files>
  <action>
    Define the ILP formulation for cost-optimal expression extraction, tailored to the custom solver.

    **Part A: Extraction DAG construction**

    Define how to build the extraction DAG from RelatedEGraph state:
    - Nodes = eclasses (canonical Values)
    - Hyperedges = enodes: a `(table_id, row_index)` in eclass `e` with child eclasses `c1..ck`
    - Source: use `reverse_index` + `eclass_enodes` to enumerate all enodes per eclass
    - Target: given a root expression (via `evaluate_expr`), extract the subgraph of reachable eclasses via BFS/DFS from root
    - Handle: literal-typed children (not eclasses — fixed cost, no decision variable, contribute only constructor cost)

    **Part B: ILP variables mapped to Quine types**

    ```
    x_{e,n} ∈ {0,1}  — 1 if enode n is selected in eclass e
        e: Value (canonical eclass ID, from union_find.find())
        n: (TableId, RowIndex)  —  enode reference, as stored in reverse_index

    y_e ∈ {0,1}  — 1 if eclass e is "active" (any selected enode has it as a child)
        e: Value (canonical)
    ```

    Note: y_e variables enable CSE — if two parents share child eclass e, y_e = 1 and e's cost is counted once.

    **Part C: Constraints (with exploitation notes)**

    1. **Root selection**: For root eclass r:
       ```
       sum_{n ∈ enodes(r)} x_{r,n} = 1
       ```
       (Must pick exactly one enode at root.)

    2. **Enode → child consistency**: For each enode n = (tid, ridx) in eclass e, with child eclasses children(n) = [c1..ck]:
       ```
       x_{e,n} ≤ y_{c_i}   for each child eclass c_i
       ```
       (If enode is selected, all its child eclasses must be active.)
       Exploitation: These constraints are extremely sparse — each involves exactly 2 variables. The constraint matrix has O(|E| * avg_enodes * avg_arity) rows but each row has only 2 nonzeros.

    3. **Activation → selection**: For each eclass e:
       ```
       y_e ≤ sum_{n ∈ enodes(e)} x_{e,n}
       ```
       (If eclass is active, at least one enode must be selected in it.)

    4. **Single enode per active eclass** (strengthened formulation, replaces 3):
       ```
       sum_{n ∈ enodes(e)} x_{e,n} = y_e
       ```
       (Exactly one enode if active, zero if inactive.)
       Exploitation: This is a generalized upper bound (GUB) constraint — one per eclass. Solvers can exploit GUB structure for efficient branching.

    5. **CSE coupling**: y_e appears in multiple parent constraints (one per parent enode referencing e). This is the ONLY coupling between otherwise-independent branches. The coupling is through y_e variables, not x variables — the x variables for different eclasses don't interact directly.

    **Part D: Objective function**

    ```
    minimize: sum_{e} sum_{n ∈ enodes(e)} constructor_cost(n) * x_{e,n}
    ```
    
    Where `constructor_cost(n)` = `cost_models[table_name.constructor_name]` (from `RelatedEGraph.cost_models`), defaulting to 0 if absent.
    
    Important: Only the constructor's OWN cost is in the objective. Child costs are NOT summed into the parent enode cost (unlike the current `compute_and_update_eclass_cost` which does `saturating_add`). Each eclass's contribution is counted exactly once through its own x_{e,n} variable.
    
    This is the key difference from the greedy cost model — the ILP naturally handles CSE because if two parents share child eclass c, the cost of c's selected enode is counted once (in x_{c,n}), not once per parent.

    **Part E: Solution extraction**

    Define how to walk the ILP solution to produce a `Term`:
    ```
    function extract_solution(solver_result, root_eclass):
        return build_term(root_eclass, solver_result.x, visited={})
    
    function build_term(eclass, x, visited):
        if eclass in visited: return Term::Cyclic  // should not happen for DAG
        visited.insert(eclass)
        (tid, ridx) = the unique enode n where x[eclass, n] = 1
        table = tables[tid]
        row = table.rows[ridx * col_count .. (ridx+1) * col_count]
        children = []
        for each child column:
            if child is Id or Name typed:
                child_eclass = union_find.find(child_value)
                children.push(build_term(child_eclass, x, visited))
            else:
                children.push(Term::Literal(child_value))
        return Term::App(constructor_name, children)
    ```

    **Part F: Complexity analysis**

    - Variables: sum_{e ∈ reachable_eclasses} |enodes(e)| + |reachable_eclasses|
    - Constraints: 1 (root) + sum_{e,n} |children(n)| (consistency) + |reachable_eclasses| (GUB)
    - Typical extraction from current examples: 20-200 eclasses, 1-5 enodes each → 40-1000 variables
    - Worst case (pathological): 1000+ eclasses with 10+ enodes each → 10000+ variables
    - Problem class: 0-1 ILP, constraint matrix is NOT totally unimodular (CSE coupling breaks total unimodularity)

    Avoid: premature optimization of the formulation. The basic formulation is correct; strengthening (lifting, cutting planes) can be added in Phase 7 if benchmarks show need.
  </action>
